## 1. Подтверждённые баги (верифицированы по коду)

### КРИТИЧНО

**K1. Удаление группы молча уничтожает абонементы, тренировки и визиты — без отката денег.**
`group.service.ts` не переопределяет удаление (наследует простой `destroy` из
`OwnedCrudService`), а FK в миграциях: `trainings.group_id → CASCADE`,
`subscriptions.group_id → CASCADE`, `visits.group_id → CASCADE`
(`20260101000002..4-create-*.js`). DELETE /groups/:id каскадом БД сносит все
тренировки группы (мимо `revertVisit` — деньги не откатываются), все абонементы
(включая активные предоплаченные с остатком занятий) и всю историю посещений.
Платежи остаются, но абонементы и визиты исчезают безвозвратно.
*Сценарий:* тренер удаляет «старую» группу → у учеников пропадают активные
абонементы и вся история.
*Фикс:* запретить удаление непустой группы (RESTRICT/проверка в сервисе) или
явное предупреждение со списком последствий + сознательный каскад через
`removeTraining`/откаты.

**K2. Списание с истёкшего по сроку абонемента.**
`subscriptions.service.ts:63` — `deduct` берёт `findAll({ studentId, isActive: true })`
и НЕ проверяет `expiresAt`. `isActive` снимается только при `remaining = 0`; по
сроку никто абонемент не деактивирует. Фронт честно показывает «Истёк срок»
(`getSubStatus`: `days < 0 → expired`), но при отметке бэк молча списывает с
протухшего абонемента вместо создания разового платежа.
*Последствие:* тренер недополучает оплату; отображение и поведение противоречат
друг другу.
*Фикс:* в `deduct`/`findActiveOrLatest` фильтровать `expiresAt >= today`
(зеркально фронтовому `getSubStatus`).

**K3. Биллинг отметки не транзакционен — возможно двойное списание.**
`training.service.ts:194-248` (`markAttendance`): порядок — `$add(attendees)` →
`deduct` → (платёж) → `visit.create`. Транзакций в бэкенде нет вообще (grep по
`transaction` — 0 вхождений). Если после `deduct` упадёт `visit.create`
(обрыв соединения/рестарт), занятие списано, а визита нет; повторная отметка не
найдёт `existing` визит и спишет **второй раз**.
*Фикс:* обернуть цепочку одного студента в `sequelize.transaction()`.

### ВАЖНО

**B1. IDOR в link-payment: чужой абонемент можно пометить оплаченным.**
`student.controller.ts:115-126` проверяет владение учеником, но `subId` НЕ
сверяется со списком абонементов этого ученика (в соседних `extend`/`deduct` —
сверяется), а `subscriptions.service.linkPayment` обновляет `where { id: subId }`
глобально. `dto.paymentId` не проверяется ни на владение, ни на существование.
*Фикс:* `student.subscriptions?.find(s => s.id === subId)` + проверка платежа
через `paymentsService.findOneForUser`.

**B2. Копирование тарифов: запись в чужую локацию + задвоение.**
`pricing-rules.service.ts:73-97` (`copy`): `toLocationId` не проверяется на
владение (можно насыпать свои тарифы в чужую локацию по UUID); повторный вызов
копирования создаёт дубликаты (нет UNIQUE-ограничения на
(userId, locationId, lessonKind, format, durationMinutes, sessionsCount) и нет
проверки существующих) → `findMatch` для авто-оплаты становится недетерминированным.
*Фикс:* проверка владения `toLocationId`; upsert/пропуск существующих; UNIQUE-индекс.

**B3. Гонка в deduct — неатомарное `remaining -= 1`.**
`subscriptions.service.ts:72-77`: read-modify-write без блокировки/атомарного
UPDATE. Два параллельных запроса отметки спишут одно занятие вместо двух (или
наоборот, по таймингу).
*Фикс:* `UPDATE ... SET remaining = remaining - 1 WHERE id = ? AND remaining > 0`
или транзакция с `lock: UPDATE` (закрывается вместе с K3).

**B4. «Отметить сегодня»: ошибка сети посередине сохранения теряется.**
`use-mark-today.ts` (`save`): `try { циклы markAttendance/removeAttendee;
invalidate; toast } finally { setSaving(false) }` — блока `catch` НЕТ. Ошибка на
середине цикла = unhandled rejection: ни тоста, ни инвалидации кэшей (они внутри
`try` после циклов), модалка остаётся открытой, часть отметок уже применена, UI
показывает старое состояние.
*Фикс:* `catch` с тостом ошибки + перенести инвалидации в `finally`.

**B5. Отрицательные суммы проходят валидацию.**
`create-payment.dto.ts:36-37` и `create-hall-cost.dto.ts:48-49`: `@IsNumber()`
без `@Min(0)` на `clientAmount`/`hallAmount` — платёж на −1000 искажает статистику.
*Фикс:* `@Min(0)` (в pricing-rules DTO уже так).

**B6. HALL_TYPE_OPTIONS неполный — 90-минутные расходы зала не редактируются.**
`payment-form-config.ts`: в опциях зала нет `single_individual_90`,
`individual_sub_4_90`, `individual_sub_8_90` (в клиентских опциях они есть;
`single_pair_90` в зале есть). Авто-оплата такие расходы зала СОЗДАЁТ
(`hallPaymentType = pricing.paymentType`), а в форме выбрать/корректно
отредактировать их нельзя.
*Фикс:* добавить три опции.

**B7. Висячие ссылки «по дизайну» (нет FK).**
`visits.subscription_id/payment_id` (миграция 20260610000001 — без references),
`subscriptions.fin_payment_id`, `payments.student_id` (allowNull, без FK).
Код устойчив (revertVisit глотает удалённый платёж `.catch()`, `restoreById`
возвращает null), но: платёж удалили вручную в Финансах → абонемент остаётся
«оплаченным» с висячим `finPaymentId`; ученика удалили → его платежи остаются
без имени.
*Фикс:* FK `ON DELETE SET NULL` + при обнулении `finPaymentId` абонемент снова
попадает в «Ожидают оплаты» (поведение согласовать).

## 3. UX-ревью: что сделать симпатичнее и удобнее

### Quick wins (дёшево, заметно)

1. **Звёздочки обязательных полей**: `Form.Item required` в формах ученика,
   группы, занятий, платежа — сейчас обязательность не видна до сабмита.
2. **Поиск в списках выбора учеников** (модалки «Добавить на тренировку»,
   «Отметить сегодня» при 30+ учениках): `Input`-фильтр над списком.
3. **Календарь на мобильном**: `min-height: 36px` у блоков (тап-таргет) +
   `title`-тултип с временем на коротких блоках, где подпись скрыта.
4. **Адаптивные модалки/Drawer**: `width: min(620px, 95vw)` у платёжной модалки,
   `width: 100%` у StudentDrawer на <768px.
5. **`inputMode="decimal"`** на денежных полях — цифровая клавиатура на телефоне.
6. **Скелетоны** (`antd Skeleton`) на первую загрузку списков/календаря вместо
   пустого экрана.
7. **«Ожидают оплаты»**: при нуле показывать «✓ Всё оплачено» (disabled), а не
   прятать кнопку — исчезновение выглядит как пропажа функции.
8. **Порядок KPI на главной**: проблемные первыми («Нужно продлить» → «Заканчиваются»
   → «Тренировок за месяц» → «Активных учеников») — тренер сперва видит, где беда.
9. **Контраст `--text-muted`** ниже WCAG AA в обеих темах — поднять непрозрачность.
10. **Подтверждения удаления**: везде называть объект («Удалить группу „Бокс“?
    Вместе с ней удалятся N тренировок и история посещений») — связано с багом K1.

## 4. Отклонённые находки агентов (проверены — НЕ баги)

- «`formatDayOfWeek`/`dayIndexOf` нужен суффикс Z» — наоборот: парсинг
  `T00:00:00` без Z даёт локальную полночь, `getDay()` корректен в любом поясе.
- «Квартал статистики ломается на границе года» — JS `Date` нормализует
  отрицательные месяцы (`new Date(2026, -3, 1)` = окт 2025), границы и подпись верны.
- «Создание занятия не инвалидирует кэш тренировок» — `useCreateTraining`
  инвалидирует `trainingKeys.all` внутри хука.
- «Переименование группы ломает связи» — в БД везде UUID, имя мапится свежим
  `group-map`; переименование безопасно.
- «Нет парных типов в SubscriptionType» — парных абонементов нет by design
  (парное занятие всегда оплачивается разовым платежом).
- «Флаг `running` воркера зависнет при падении» — есть `try/finally`.
- «`$add(attendees)` дублирует связи» — повторный `$add` для M:N безопасен,
  идемпотентность обеспечена проверкой существующего визита (кроме сценария K3).
