# Еженедельная серия независимо от абонемента (Часть 1) — дизайн

**Дата:** 2026-06-07
**Статус:** утверждён дизайн, ожидает вычитки спецификации
**Ветка:** `feat/google-calendar-sync` (работа продолжается на ней)

## 1. Цель

Дать тренеру создавать **еженедельную серию индивидуальных занятий, не привязанную
к абонементу** — для клиентов, которые ходят каждую неделю, но платят разово за
каждое занятие. Сейчас длина серии жёстко равна числу занятий в абонементе, а выбор
абонемента обязателен; при разовом абонементе «повторять» молча создаёт одно занятие.

Это **Часть 1** из двух. Часть 2 (отдельная спека) — авто-оплата разового занятия при
отметке посещения. Часть 1 самодостаточна: создаёт плановые занятия; оплата пока
заносится вручную через финансы.

## 2. Решения (итог брейншторма)

| Вопрос | Решение |
|---|---|
| Длина серии | Явное поле «повторить N раз» — всегда источник длины (а не абонемент). |
| Абонемент | Опционален: переключатель оплаты «Абонемент / Разово». |
| Оплата (разово) | При создании — ничего. Все занятия плановые; оплата за раз — при проведении (Часть 2). |
| Режим «Разово» | **Все** занятия серии плановые (включая первое сегодняшнее), без «провести сейчас». |
| N при абонементе | По умолчанию = размеру абонемента, но редактируемо. |

## 3. Что НЕ входит (Часть 2 и далее)

- Авто-запись разового платежа при отметке посещения (отдельная спека — Часть 2).
- Изменение поведения групповых тренировок (у них «повторять» нет; не трогаем).
- «Повторять до даты» и бессрочные серии (выбрана модель «N раз»).

## 4. Форма и UX (`individual-session-modal.tsx`)

Две независимые оси (сейчас спутаны):

1. **Оплата** — переключатель «Абонемент | Разово»:
   - *Абонемент* — как сейчас: выбор типа/слота, создаётся абонемент + авто-платёж.
   - *Разово* — селекторы типа/слота скрыты; абонемент не создаётся.
2. **Повторение** — галочка «повторять еженедельно» → числовое поле «повторить N раз».

4 комбинации: Абонемент×(одно/серия), Разово×(одно/серия).

Поле N: целое ≥ 1, максимум 52. Если «повторять» снято — серия из 1 (N=1). В режиме
«Абонемент» N инициализируется размером абонемента (`SUB_TYPE_TOTALS[subType]`), но
редактируемо.

## 5. Логика создания (`use-individual-session.ts`)

Длина серии **всегда из N**. Даты: `weeklySeriesDates(date, N)` = `date, date+7, …,
date+7·(N−1)`.

**Режим «Абонемент»** (поведение как сейчас, меняется только источник количества):
- Создаётся абонемент (если нет активного) + авто-платёж (`autoCreatePayment`).
- Первое занятие: участник списывается, если дата сегодня/в прошлом; иначе плановое.
- Будущие занятия серии: плановые (`plannedStudentId`, без списания).
- Количество будущих = `N − 1` (раньше было `totalSessions − 1`).

**Режим «Разово»**:
- Абонемент и авто-платёж **не создаются**.
- **Все** N занятий — плановые (`plannedStudentId`, `attendees: []`), без списания.
- Сегодняшнее плановое занятие попадёт в «Отметить за сегодня» и проводится там.

Общие для обоих режимов:
- Все занятия серии получают общий `recurringId` (uuid) + `recurring=true`.
- `isPrime` каждого занятия считается по его собственной дате: `isPrimeTime(date_i, time, location)`.
- Каждое занятие со временем ставит `upsert` в Google (через `createTraining` →
  бэкенд `enqueueUpsert`). Итог: в Google появляются все N событий — это закрывает
  исходную проблему «серия не дублируется в календарь».

## 6. Граничные случаи

- **Конфликты:** проверяется только первое занятие (как при текущем создании); будущие
  не проверяются (согласуется с существующим рекуррентным созданием).
- **N=1:** одно занятие (плановое в «Разово», обычное в «Абонементе»).
- **Прошедшая дата старта:** допускается, спец-обработки нет.
- **Совместимость с редактированием серии:** общий `recurringId` означает, что уже
  реализованное редактирование «вся серия» применимо к этим занятиям.

## 7. Чистая функция дат

Генерацию дат серии (шаг по неделям, границы месяца) выносим в именованный
фронт-хелпер ради ясности и переиспользования:

- Создать `apps/frontend/src/entities/trainings/model/weekly-series.ts`:
  ```ts
  /** N дат с недельным шагом от стартовой (ISO YYYY-MM-DD), включая старт. */
  export function weeklySeriesDates(startISO: string, count: number): string[]
  ```
  Реализация на UTC-датах (`new Date(startISO + 'T00:00:00Z')`, `setUTCDate(+7·i)`,
  срез `toISOString().slice(0,10)`), без сдвига часового пояса.

Фронт использует `weeklySeriesDates(date, N)` вместо ручного `setDate(+7*w)`.

> **Примечание по тестам:** у фронтенда нет тест-раннера, а у пакета `shared` —
> тоже (только сборка), поэтому хелпер живёт во фронте и проверяется сборкой +
> ручным кейсом с границей месяца (см. §9). Это согласуется со всем остальным
> фронт-кодом проекта (логика без юнит-тестов, проверка через `tsc` + ручной прогон).

## 9. Тестирование

- **Сборка (тип-чек):** `npm run build -w apps/frontend` — без ошибок типов.
- **Ручное:**
  - «Разово» серия N=4 → 4 плановых занятия в расписании + 4 события в Google.
  - **Граница месяца:** старт на `2026-06-29`, N=3 → даты `06-29`, `07-06`, `07-13`
    (проверка корректного шага через смену месяца).
  - «Абонемент» серия → как раньше, но N редактируемо (4/8 по умолчанию).
  - «Разово» одно (N=1) → одно плановое занятие; деньги не трогаются.
  - Сегодняшнее плановое из «Разово» серии видно в «Отметить за сегодня».

## 10. Открытые ограничения (осознанно)

- Авто-оплата при отметке — в Части 2 (сейчас разовый платёж заносится вручную).
- Нет проверки конфликтов для будущих занятий серии.
- Групповые тренировки не затрагиваются.
