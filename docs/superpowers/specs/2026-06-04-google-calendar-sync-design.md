# Синхронизация расписания с Google Календарём — дизайн (Фаза 1)

**Дата:** 2026-06-04
**Статус:** утверждён дизайн, ожидает вычитки спецификации
**Ветка:** `feat/google-calendar-sync` (от `origin/main` @ 069e078)

## 1. Цель

Дать каждому тренеру одностороннюю выгрузку своего расписания в его Google Календарь.
Внёс / изменил / удалил тренировку в системе → событие появляется / меняется /
удаляется в Google. Источник правды — всегда база TriKick; Google только отражает.

Инструмент для **тренеров** (видеть стабильное расписание у себя в телефоне/Google),
а не для клиентов.

## 2. Решения (итог брейншторма)

| Вопрос | Решение |
|---|---|
| Направление | Односторонняя: система → Google. Обратной синхронизации нет. |
| Охват аккаунтов | Каждый тренер подключает **свой** Google (per-user OAuth). |
| Целевой календарь | При подключении показываем список существующих календарей тренера + пункт «➕ Создать новый „TriKick"». Тренер выбирает: основной / существующий / создать новый. |
| Что синхронизируем | **Всё расписание**: групповые + индивидуальные + плановые будущие. |
| Часовой пояс | Берём из настроек Google-аккаунта тренера (а не из локаций). |
| Повторяющиеся | **Одно событие на каждое занятие** (1:1 с базой), без нативного RRULE. |
| Надёжность | Через **аутбокс-очередь** + фоновый воркер (`@nestjs/schedule`), без Redis. |
| Нейминг | Остаётся **TriKick** (переименование в TrickCoin не применяем). |

## 3. Что НЕ входит в Фазу 1

- Обратная синхронизация (правки из Google → в систему).
- Apple Calendar (это **Фаза 2** — универсальная подписка ICS, см. §10).
- Клиентская функция «покажи мои свободные окна» (экспорт/картинка) — отдельно, позже.
- Нативные повторяющиеся события (RRULE) с исключениями.
- Ссылки видеоконференций (Google Meet) в событиях.

## 4. Архитектура

Новый изолированный модуль `CalendarModule` (backend). `TrainingService` при
мутациях лишь **кладёт задачу в аутбокс** — он не знает деталей Google. Фоновый
воркер внутри `CalendarModule` разбирает очередь и общается с Google API.

```
TrainingService (create/update/delete)
        │  enqueue(task)
        ▼
calendar_sync_tasks (аутбокс в БД)
        │  каждые ~20–30 c
        ▼
CalendarSyncWorker  ──►  Google Calendar API
        │
        └─ пишет google_event_id обратно в training
```

Чтобы избежать циклической зависимости: `CalendarModule` подключает **модель**
`Training` через `SequelizeModule.forFeature([Training])` (не сервис), а
`TrainingModule` импортирует `CalendarModule` ради лёгкого `enqueue`. Воркер читает
данные тренировки сам по `trainingId`.

## 5. Данные (новое в БД)

### 5.1 Идентификатор события (детерминированный, без колонки)
- Id события в Google **выводим из id тренировки**: `eventId = trainingId` без
  дефисов в нижнем регистре (32 hex-символа — валидный id для Google Calendar).
- Это делает `upsert`/`delete` идемпотентными (нет гонок, не нужно писать id
  обратно в БД) и **убирает отдельную колонку `google_event_id`**. Одна
  тренировка → одно событие (тренировка принадлежит одному `user_id`, у каждого
  тренера свой календарь).

### 5.2 Таблица `calendar_connections`

|---|---|---|

| `provider` | STRING | `google` (на будущее — расширяемо) |
| `refresh_token_enc` | TEXT | refresh-токен, **зашифрован** ключом из env |

## 6. Подключение Google (OAuth)

Scope: `https://www.googleapis.com/auth/calendar` (управление календарями и
событиями — покрывает список, создание календаря и события). `access_type=offline`
+ `prompt=consent` — чтобы получить refresh-токен.

Эндпоинты нового `CalendarController` (под общим префиксом `/api`):
- `GET /calendar/google/auth-url` → URL согласия Google (`state` = подписанный `userId`).
- `GET /calendar/google/callback?code&state` → обмен кода на токены, шифруем и
  сохраняем refresh-токен, тянем список календарей + пояс аккаунта, статус
  `connected` (но `calendar_id` ещё не выбран).
- `GET /calendar/calendars` → список календарей тренера для пикера.
- `POST /calendar/select` → `{ calendarId }` **или** `{ create: true, name }`; при
  `create` создаём календарь; сохраняем `calendar_id` + пояс; запускаем бэкфилл.
- `POST /calendar/resync` → ставит в аутбокс `upsert` всех будущих тренировок.
- `POST /calendar/disconnect` → статус `disconnected`, чистим токен; события
  оставляем (см. §8).
- `GET /calendar/status` → состояние подключения для экрана настроек.

**Безопасность:** refresh-токен шифруется ключом `CALENDAR_TOKEN_ENC_KEY` (AES,
из env). Client ID/Secret и redirect URI — тоже из env (см. §9).

## 7. Содержимое события

- **Заголовок:** групповая — название группы; индивидуальная — имя ученика (из
  `plannedStudentId` или первого посетителя).
- **Время:** `start.dateTime` = `date` + `time` в поясе аккаунта; `end` =
  start + `sessionDuration` минут; `timeZone` = `calendar_time_zone`.
- **Место (`location`):** название + адрес локации (для онлайна — пометка «Онлайн»).
- **Описание:** заметка тренировки + флаги (онлайн / прайм).
- **Привязка:** `extendedProperties.private.trainingId = <id>` — надёжная связь
  события с тренировкой (помимо `google_event_id` в БД).

## 8. Точки врезки и поведение

| Действие в системе | Эндпоинт | Что в аутбокс |
|---|---|---|
| Создание тренировки | `POST /trainings` | `upsert` |
| Изменение | `PATCH /trainings/:id` | `upsert` |
| Удаление | `DELETE /trainings/:id` | `delete` (снимок `calendarId`+`eventId`) |
| Удаление серии | `DELETE /trainings/recurring/:recurringId` | `delete` на каждое занятие |
| Отметка/снятие посещения | `.../attendees…` | `upsert` (обновить заголовок/статус) |

Плановые будущие занятия тоже улетают (часть расписания). При отмене занятия —
событие удаляется.

**Обработка `upsert` воркером:** загрузить `Training` по id; нет (удалена) —
пропустить. Событие создаётся/обновляется по детерминированному id (см. §5.1):
`events.insert`, а при конфликте `409` (уже есть) → `events.patch`. **`delete`:**
`events.delete` по тому же выведенному id; `404/410` (события нет) — считаем
успехом. Тренировка без времени → ставим `delete` (убрать возможное устаревшее
событие).

## 9. Надёжность и крайние случаи

- **Воркер:** `@nestjs/schedule` `@Interval` (~20–30 c) берёт пачку `pending` с
  `run_after <= now`. Успех → `done` (+ сохранить `event_id`). Ошибка → `attempts++`,
  экспоненциальный бэкофф через `run_after`; после N попыток → `failed` с
  `last_error` (без бесконечного долбления). Токен обновляется автоматически
  (googleapis из refresh-токена).
- **Токен отозван / протух безвозвратно:** статус `needs_reconnect`, в настройках —
  баннер «нужно переподключить». Тренировки спокойно сохраняются дальше.
- **Бэкфилл:** при выборе календаря и по кнопке «Синхронизировать всё» — `upsert`
  будущих тренировок (прошлые — по желанию, по умолчанию не трогаем).
- **Тренировка без времени (`time = ''`):** не синхронизируем — у события нет
  корректного начала (решение подтверждено).
- **Отключение:** события в Google остаются. Если календарь «TriKick» создавали мы —
  предложим отдельной кнопкой «удалить календарь целиком».

## 10. Зависимости и конфигурация

- npm: `googleapis`, `@nestjs/schedule` (добавить в backend).
- `ScheduleModule.forRoot()` + `CalendarModule` в `app.module`.
- Новые env (+ `env.validation`):
  - `GOOGLE_OAUTH_CLIENT_ID`
  - `GOOGLE_OAUTH_CLIENT_SECRET`
  - `GOOGLE_OAUTH_REDIRECT_URI`
  - `CALENDAR_TOKEN_ENC_KEY`
- Миграции: `google_event_id` в `trainings`; таблицы `calendar_connections`,
  `calendar_sync_tasks`.
- **Вне кода (от пользователя):** проект в Google Cloud Console, включённый Calendar
  API, OAuth-клиент (Client ID/Secret), настроенный redirect URI. Пошаговую
  инструкцию выдам на этапе реализации.

## 11. Тестирование

- **Юнит:** маппинг `Training → событие Google`; логика бэкоффа; решение
  insert/update по `google_event_id`.
- **Интеграция:** постановка в аутбокс при create/update/delete; обработка воркером
  с замоканным Google-клиентом.
- **Ручное (API/реальный тест-аккаунт):** полный цикл подключения; create/update/
  delete события; бэкфилл; сценарий отозванного токена → `needs_reconnect`.

## 12. UI (frontend)

- Экран настроек: блок «Google Календарь» — состояние (не подключено / подключено к
  «<имя календаря>» / нужно переподключить), кнопки **«Подключить»**,
  **«Синхронизировать всё»**, **«Отключить»**.
- После согласия Google — модалка выбора календаря: список существующих + «➕ Создать
  „TriKick"».
- Баннер «нужно переподключить» при `needs_reconnect`.
