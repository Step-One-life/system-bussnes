# Редактирование тренировки — дизайн

**Дата:** 2026-06-06
**Статус:** утверждён дизайн, ожидает вычитки спецификации
**Ветка:** `feat/google-calendar-sync` (работа продолжается на ней)

## 1. Цель

Дать тренеру возможность **редактировать существующую тренировку** через UI: менять
дату, время, локацию, заметку, а также флаги «онлайн» и «прайм». Сейчас такой кнопки
нет — поменять время занятия можно только пересозданием. Бэкенд (`PATCH /trainings/:id`)
и фронтовый слой API (`updateTraining`) это уже умеют, не хватает формы и точек входа.

Правки корректно переносят занятие, проверяются на конфликты расписания и
автоматически синхронизируются в Google Календарь (через уже работающий аутбокс-воркер,
см. `2026-06-04-google-calendar-sync-design.md`).

## 2. Решения (итог брейншторма)

| Вопрос | Решение |
|---|---|
| Объём полей | Дата, время, локация, заметка + тумблеры «онлайн» и «прайм». Группу/тип и состав учеников **не трогаем** (меняются существующими кнопками). |
| Подход | Отдельная единая модалка `EditTrainingModal` для всех типов (Подход A). Форму создания не трогаем. |
| Точки входа | Обе: кнопка ✏️ в карточке списка **и** кнопка «Редактировать» в `CalendarTrainingModal`. Обе открывают одну форму. |
| Повторяющиеся серии | При сохранении спрашивать «только это занятие / вся серия» (как в Google Calendar). |
| Дата в серии | Дата **никогда** не распространяется на серию — только на открытое занятие. |

## 3. Что НЕ входит

- Редактирование группы/типа тренировки и состава учеников (есть отдельные кнопки).
- Per-occurrence проверка конфликтов при массовом обновлении серии (см. §6).
- Запрет редактирования прошедших тренировок (намеренно разрешаем).
- Drag-and-drop переноса в календаре (возможно, отдельная задача позже).

## 4. UX и точки входа

Новая модалка **`EditTrainingModal`** (заголовок «Редактировать занятие»), единая для
групповых / индивидуальных / онлайн тренировок (поля одинаковы, т.к. состав учеников и
группа не редактируются).

**Точки входа (обе ведут в одну форму):**
- **Список:** в развёрнутой карточке `training-item.tsx` — иконка-кнопка ✏️ рядом с «Удалить».
- **Календарь:** клик по событию уже открывает `CalendarTrainingModal`; добавляем туда
  кнопку «Редактировать». Клик по блоку календаря **не перехватываем** — текущее
  поведение сохраняется.

Состояние входа — в `use-trainings-page.ts`: `editTarget: Training | null` +
`openEditTraining(training)` / закрытие. Модалка рендерится на странице:
`<EditTrainingModal open={!!editTarget} training={editTarget} … />`.

## 5. Поля, конфликты, онлайн/прайм

**Редактируемые поля:** `date`, `time`, `locationId`, `note`, `isOnline`, `isPrime`.

- **Предзаполнение** из переданного `training`.
- **Конфликты:** переиспользуем `checkTrainingConflict`, расширив сигнатуру опциональным
  `excludeTrainingId`, чтобы занятие не конфликтовало само с собой. Показ через
  существующий `ConflictHint`; при конфликте сохранение блокируется (как в создании).
- **Прайм:** при смене даты/времени/локации показываем `PrimeHint` (автоподсказка от
  `isPrimeTime`), но явный тумблер даёт ручной оверрайд. В БД пишется значение тумблера.
- **Онлайн:** тумблер `isOnline`. В событии Google это уже отражается (`event-builder`
  ставит место «Онлайн»).

## 6. Повторяющиеся серии

Поведение при «Сохранить» зависит от наличия `recurringId`:

- **Серии нет** → `PATCH /trainings/:id`, без вопросов.
- **Серия есть** → диалог выбора (как Google Calendar):
  - **«Только это занятие»** → `PATCH /trainings/:id`.
  - **«Вся серия»** → `PATCH /trainings/recurring/:recurringId` (массовое обновление).

**Что распространяется на серию:**

| Поле | «Только это» | «Вся серия» |
|------|:---:|:---:|
| Время | ✅ | ✅ на все |
| Локация | ✅ | ✅ на все |
| Заметка | ✅ | ✅ на все |
| Онлайн | ✅ | ✅ на все |
| Прайм | ✅ | ✅ на все (пересчёт по дате каждого занятия) |
| **Дата** | ✅ | ⛔ только открытое занятие |

**Дата принципиально не распространяется на серию** — у каждого занятия своя дата. Если
в режиме «вся серия» изменена и дата, она применяется только к открытому занятию,
остальные поля — ко всем. В форме показываем подсказку об этом.

**Конфликты при «вся серия»:** проверяем только открытое занятие; per-occurrence
проверку при массовом применении не выполняем (осознанное ограничение первой версии).

## 7. Бэкенд и синхронизация с Google

**Изменения на бэкенде:**
- **Новый эндпоинт** `PATCH /trainings/recurring/:recurringId` → `updateSeriesForUser(userId,
  recurringId, data)`: проходит по всем занятиям серии пользователя, применяет
  распространяемые поля (время/локация/заметка/онлайн/прайм; **не дату**) и на каждое
  ставит `upsert`. Симметрично существующему `DELETE /trainings/recurring/:recurringId`.
- `PATCH /trainings/:id` (одиночное) — уже существует (`updateForUser`), не меняем.

**Синхронизация** переиспользует уже проверенный механизм:
- Каждое обновление (одиночное или per-occurrence) ставит `upsert` → воркер делает `patch`
  события по детерминированному id (от id тренировки) → **то же событие двигается/
  обновляется, без дублей**. Идемпотентно (insert→409→patch).

## 9. Тестирование

- **Backend (jest, как сейчас — чистые функции/сервисы):**
  - `updateSeriesForUser` применяет распространяемые поля ко всем занятиям серии и ставит
    `upsert` на каждое; дата на серию не распространяется.
  - Одиночный `updateForUser` по-прежнему ставит `upsert` (регресс не задет).
  - `checkTrainingConflict` исключает занятие по `excludeTrainingId` (чистая функция).
- **Ручное:**
  - Правка времени одиночной тренировки → событие переехало в Google.
  - «Вся серия» → переехали все занятия серии; дата изменила только открытое.
  - Тумблеры «онлайн»/«прайм» отражаются и в приложении, и в событии Google.
  - Конфликт расписания блокирует сохранение; занятие не конфликтует само с собой.

## 10. Открытые ограничения (зафиксированы, осознанно)

- Нет per-occurrence проверки конфликтов при «вся серия».
- Нет drag-and-drop переноса в календаре.
- Редактирование прошедших тренировок разрешено (upsert корректно отработает).
