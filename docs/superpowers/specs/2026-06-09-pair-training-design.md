# Парные (сплит) тренировки — дизайн

**Дата:** 2026-06-09
**Статус:** утверждён дизайн, ожидает вычитки спецификации
**Ветка:** `feat/pair-training`

## 1. Цель

Дать возможность записывать **парные (сплит) тренировки** — занятие на **ровно двух**
учеников с отдельной (обычно меньшей) стоимостью. Цена и расход зала задаются **на одного
участника**; каждый ученик платит свою сумму. Парная может быть как разовой, так и
**еженедельной серией** (некоторые пары ходят регулярно). Оплата фиксируется **при отметке
посещения** каждого ученика (как авто-оплата Части 2).

## 2. Решения (итог брейншторма)

| Вопрос | Решение |
|---|---|
| Цена | Тариф «парная» задаёт цену **с одного** ученика; каждый платит свою → два платежа. |
| Число участников | **Ровно 2.** |
| Разовая/абонемент | Только **разовая** (формат `single`); парных абонементов нет. |
| Повтор | Можно **еженедельной серией** (переиспользуем механику серий). |
| Момент оплаты | При **отметке посещения** каждого (механика Части 2). |
| Расход зала | **С каждого** участника (в тарифе — доля на одного; двое = полный зал). |
| UI | Отдельная **кнопка «Парная»** в выборе типа занятия. |

## 3. Что НЕ входит (YAGNI)

- Парные абонементы (4/8 парных занятий).
- 3+ участников (строго 2).
- Парный онлайн.
- Деление одной общей цены между участниками (цена всегда «за ученика»).

## 4. Тарифы и типы оплаты

- `LessonKind` += `'pair'` — в `packages/shared/src/enums/index.ts` и
  `apps/frontend/src/entities/finance/model/types.ts`.
- `ClientPaymentType` += `'single_pair'`, `'single_pair_90'` (там же; `HallPaymentType` = `ClientPaymentType`).
- Форма тарифа (`entities/finance/pricing/pricing-config.ts` + локали): новая опция вида
  занятия «Парная». Тариф: `lessonKind='pair'`, `format='single'`, `durationMinutes` 60/90,
  `sessionsCount=1`. `client_price`/`client_prime_price` и `hall_cost`/`hall_prime_cost` —
  на одного участника.
- Резолв тарифа (`entities/finance/lib/pricing-lookup.ts`): расширить `subTypeToTuple` и
  `subPaymentType`/`clientTypeToTuple` внутренними под-типами `'1_pair'` → кортеж
  `pair/single/60/1`, `'1_pair_90'` → `pair/single/90/1`; `subPaymentType('1_pair') →
  'single_pair'`, `('1_pair_90') → 'single_pair_90'`. Маппинг FIN_LABELS для новых типов.

## 5. Модель тренировки (бэкенд)

- Миграция `trainings`: добавить `is_pair` (BOOLEAN, NOT NULL, default 0) и
  `planned_student_id_2` (CHAR(36), NULL).
- `training.model.ts`, `dto/create-training.dto.ts`: добавить `isPair`, `plannedStudentId2`.
- `training.service.createTraining` / `createForUser`: пробрасывать оба поля.
- Поля серии (`recurring`, `recurringId`) уже есть — переиспользуются для парных серий.
- Серверный барьер наслоения (`findOverlaps`) работает без изменений: парное — одно занятие
  с одним временем.

## 6. Создание (фронт)

- В выборе типа занятия (`entities/trainings/form/training-type-modal.tsx`) добавить кнопку
  **«Парная»** рядом с «Индивидуальная»/«Online».
- Парное занятие привязано к **индивидуальной группе-контейнеру** (как индивидуальное/
  online занятие); группа берётся из контекста выбора типа так же, как для individual/online.
- Новая модалка `pair-session-modal.tsx` + хук `use-pair-session.ts` (по образцу
  `use-individual-session` в режиме «Разово», но проще — без абонемента/выбора оплаты):
  - два селекта клиента (нельзя выбрать одного и того же),
  - дата, время, длительность (1ч/1.5ч), локация,
  - переключатель «Еженедельно» + поле «Сколько занятий» (как в индивидуальной).
- Создаёт занятия плановыми (`is_pair=true`, оба плановых ученика, `attendees=[]`):
  - разовая → одно занятие;
  - серия → `weeklySeriesDates(date, N)` занятий с общим `recurringId`.
- Проверка конфликта расписания **по каждой дате серии** (как в индивидуальной — через
  `checkTrainingConflict`), блокировать с перечислением конфликтных дат.

## 7. Отметка и оплата

- Парное занятие показывается в «Отметить за сегодня» (`pages/home/use-mark-today.ts`,
  блок индивидуальных) и в календаре как плановое с **двумя** учениками (источник —
  `plannedStudentId` + `plannedStudentId2`).
- При отметке ученика — `markAttendanceWithPayment` (`entities/trainings/model/training-logic.ts`):
  если `training.isPair`, использовать под-тип `'1_pair'`/`'1_pair_90'` (по `sessionDuration`)
  вместо индивидуального, чтобы `autoCreatePayment` резолвил **pair**-тариф. Остальное — как
  в Части 2: нет тарифа → тихий пропуск; онлайн неактуален (парная очная).
- **Важно — не смешивать с абонементами:** для парного занятия отметка **НЕ списывает**
  индивидуальный абонемент, даже если он активен у ученика на эту группу, и **всегда** пишет
  парный платёж (игнорируем проверку `hasActive`). Реализовать через путь добавления
  посещения **без списания** (на бэке уже есть `attachAttendeesWithVisits` — отметка без
  `deduct`; для парного занятия использовать его вместо обычного `markAttendance`/`deduct`).
- Двое отмечены → два платежа `single_pair` + два расхода зала (= полный зал). Защита от
  дублей — та же (платёж на переход «не был → отмечен»).

## 8. Отображение

- В календаре/списке (`entities/trainings/calendar`, карточка занятия) парное занятие
  подписывать как «Парная: {ИмяА} + {ИмяБ}». Признак — `is_pair`.

## 9. Тестирование

Бэкенд: сборка + существующие тесты. Фронт: `npm run build -w apps/frontend`. Ручной прогон:
- Создать парную (разовую) на двух учеников без pair-абонемента → занятие плановое.
- Отметить обоих → в «Финансах» два платежа `single_pair` + два расхода зала.
- Отметить одного → один платёж; повтор сохранения дубль не создаёт.
- Создать парную **серию** (например 4 недели) → 4 плановых парных занятия с общим
  `recurringId`; конфликт по будущей дате блокирует с перечислением дат.
- Нет pair-тарифа на локации → отметка прошла, платёж не записан, без ошибок.

## 10. Осознанные ограничения

- Снятие отметки не откатывает платёж (ручная корректировка — как в Части 2).
- Если у одного из пары есть активный индивидуальный абонемент на эту группу — он НЕ
  списывается (парная — отдельный разовый тип); пишется парный платёж. (Парные и абонементы
  не смешиваются by design.)
