# Онбординг тренера — дизайн

Дата: 2026-06-16 · Ветка: `feat/trainer-onboarding`

## 1. Цель

Сейчас после регистрации тренер попадает на пустую главную без проводника: пустая лента,
нулевые KPI, кнопки есть, но непонятно, с чего начать (`register-page` делает
`navigate('/')` без какого-либо сетапа). Задача — дать мягкий проводник «с чего начать»,
который доводит тренера до момента ценности (первое записанное занятие), но не блокирует
и не навязывает. Бэкенд **не меняется** (как в проектах «Мой день» и «Журнал v1»).

## 2. Решение (гибрид B+C)

Два связанных, но независимых механизма:

- **B — чеклист «Первые шаги»** на главной: карточка со списком задач и прогрессом. Это
  «карта пути».
- **C — умные пустые экраны**: каждый ключевой пустой экран ведёт к своему действию крупной
  кнопкой (`EmptyState action`). Это «по месту».

Они работают вместе: тап по шагу чеклиста ведёт в нужный раздел, где пустой экран с CTA
(часть C) доводит до формы. Никаких отдельных «онбординг-экранов» — переиспользуем
существующие формы (`LocationFormModal`, `GroupFormModal`, `StudentFormModal`,
`TrainingTypeModal`, `RuleFormModal`).

Мастер-визард (модальный пошаговый, вариант A из брейншторма) **отклонён**: на телефоне
(80% пользователей) ощущается навязчивым и его охотно пропускают.

## 3. Чеклист «Первые шаги» — состав

Пять шагов, каждый можно пропустить:

| # | Шаг | id | ✓ когда (по реальным данным) |
|---|-----|----|------------------------------|
| 1 | Добавить локацию | `location` | есть хотя бы одна локация |
| 2 | Создать группу | `group` | есть хотя бы одна **не-индивидуальная** группа |
| 3 | Добавить ученика | `student` | есть хотя бы один ученик |
| 4 | Записать первое занятие | `training` | есть хотя бы одна тренировка |
| 5 | Настроить тариф | `pricing` | есть хотя бы одно правило тарифа |

- Шаг 4 на целевом экране даёт развилку типа занятия (групповое / индивидуальное / онлайн /
  парное) — существующий `TrainingTypeModal`. Тренер-индивидуал создаёт индивидуальное, не
  трогая шаг «Группа» (пропускает его).
- Индивидуальная группа-контейнер (`ensureIndividualGroup`) **не** засчитывает шаг «Группа» —
  только обычная группа, созданная тренером.

## 4. Логика галочек — автоматическая, по факту данных

Галочки **не** хранят отдельный «прогресс» — они вычисляются из реальных данных, которые
живут на сервере. Создал сущность где угодно (хоть мимо чеклиста) — шаг сам станет `done`.
Это значит: прогресс одинаков на всех устройствах (данные общие), чеклист — честное зеркало
состояния.

Чистая функция (под vitest), не зависит от react-query:

```
type OnboardingStepId = 'location' | 'group' | 'student' | 'training' | 'pricing'
type OnboardingStepStatus = 'done' | 'active' | 'skipped'

buildOnboardingSteps(input: {
  hasLocation: boolean
  hasGroup: boolean        // не-индивидуальная
  hasStudent: boolean
  hasTraining: boolean
  hasPricing: boolean
  skipped: OnboardingStepId[]
}): { id: OnboardingStepId; status: OnboardingStepStatus }[]
```

Статус шага: `done` если данные есть; иначе `skipped` если id в `skipped`; иначе `active`.

## 5. Состояние и видимость

Только два флага, и только про «не показывай мне» — хранятся в **localStorage** (per-user
ключ `tk_onboarding_<userId>`):

```
{ skipped: OnboardingStepId[], hidden: boolean }
```

- **Видимость карточки:** `!hidden && steps.some(s => s.status === 'active')`. Когда все
  шаги `done` или `skipped` — карточка исчезает сама.
- **«Пропустить» шаг** → добавляет id в `skipped` (шаг сереет, не торопит).
- **«Скрыть ✕»** → `hidden = true`, карточка уходит сразу.
- **Возврат:** пункт **«Первые шаги»** в меню профиля сбрасывает `hidden = false`. Если
  все шаги `done` — пункт не показываем (или показываем неактивным «всё настроено»).

**Почему localStorage, а не бэкенд:** прогресс (галочки) и так общий через серверные данные.
В localStorage — только жест «скрыл/пропустил недоделанное», который теряется при переходе на
другое устройство. Последствие: если тренер **не доделал** настройку и скрыл карточку на
телефоне, на компьютере она вернётся — но только с реально незавершёнными шагами (полезное
напоминание). Если **всё настроено** — карточки нет нигде. Ради этого редкого и безобидного
случая не городим поле в БД + миграцию + эндпоинт (YAGNI). Per-user ключ нужен, т.к.
`auth-provider` при смене аккаунта чистит кэши react-query, но **не** localStorage.

## 6. Открытие шагов

Тап по шагу ведёт тренера к созданию нужной сущности через **существующий UI** (форму/модалку).
Точная механика (открыть модалку прямо с главной vs навигация на раздел) — на этапе плана;
ориентир — навигация на соответствующий раздел, где пустой экран с CTA (часть C) доводит до
формы, чтобы не разрастался `home-page` и B↔C работали в связке. Шаг `pricing` доступен только
при наличии локации (`RuleFormModal` требует `locationId`) — иначе подсказка «сначала локация».

## 7. Часть C — умные пустые экраны

`EmptyState` уже поддерживает проп `action` (ReactNode). Добавляем CTA-кнопку в «тупиковые»
пустые состояния, где её сейчас нет:

- **Группы** (`groups-page`) → «+ Создать группу»
- **Тренировки** (`trainings-page`) → «+ Записать занятие»
- **Индивидуальные** (`individual-page`) → «+ Записать занятие»
- **«Нет учеников» в группе** (`group-detail`) → «+ Добавить ученика»
- **«Нет локаций» в тарифах** (`pricing-tab`) → «+ Добавить локацию»

Единый паттерн «крупная кнопка действия в пустом экране». Экраны, заполняющиеся сами (журнал,
лента дня), не трогаем — там пустое состояние уместно как есть.

## 8. Компоненты (фронт)

Новый модуль `entities/onboarding/`:
- `model/onboarding-steps.ts` — чистая `buildOnboardingSteps` + типы (vitest).
- `model/use-onboarding-state.ts` — хук localStorage (`skipped`, `hidden`), per-user ключ.
- `ui/onboarding-checklist.tsx` — карточка на главной (рендер шагов, прогресс, «Пропустить»,
  «Скрыть»).
- `index.ts` — публичный API модуля.

Интеграция: `home-page` рендерит `OnboardingChecklist` сверху (над лентой), когда карточка
видима. Источник флагов наличия — существующие хуки (`useLocations`, `useGroups`,
`useStudents`, `useTrainings`, тарифы — уточнить в плане: по дефолтной локации или общим
запросом). Пункт «Первые шаги» добавляется в `profile-menu`.

## 9. i18n

Новый раздел `onboarding.*` в `ru`/`en` (заголовок карточки, названия 5 шагов и подсказки,
«Пропустить», «Скрыть», «Первые шаги», прогресс с плюрализацией). Часть C — переиспользуем
существующие ключи действий, где возможно; недостающие CTA-подписи добавляем.

## 10. Тестирование

- **vitest** (чистые функции): `buildOnboardingSteps` — матрица состояний данных и `skipped`;
  логика видимости карточки.
- UI вживую — на пользователе (правило проекта: Playwright не запускаем без явной просьбы).
- Гейт: фронт build (tsc+vite), vitest, eslint 0/0.

## 11. Вне скоупа (YAGNI)

- Бэкенд-хранение состояния онбординга и синхронизация жеста «скрыл» между устройствами.
- Мастер-визард, тур по интерфейсу, всплывающие подсказки-коучмарки.
- Изменение flow регистрации (остаётся `navigate('/')`).
- Аналитика прохождения онбординга.

## 12. Ловушки

- **`training.groupId` на фронте = имя группы**, не UUID (репозиторий мапит) — для флага
  «есть не-индивидуальная группа» считать по `groups` (с `isIndividual`), не по тренировкам.
- **localStorage переживает смену аккаунта** (в отличие от кэшей react-query) — отсюда
  per-user ключ, иначе состояние «протечёт» между тренерами на одном браузере.
- **Карточка не должна мигать** при первой загрузке, пока данные ещё грузятся — показывать
  только после резолва запросов (иначе мелькнёт «0/5» и схлопнется).
