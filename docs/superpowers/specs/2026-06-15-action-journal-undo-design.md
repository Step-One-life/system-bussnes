# Журнал действий + отмена (undo) — дизайн

Дата: 2026-06-15 · Ветка: `feat/action-journal-undo`

## 1. Цель

Лента «что произошло» — отметки, списания, платежи, продления, создание занятий — с
возможностью откатить ошибочное действие одной кнопкой. Посыл «интерфейс доверия»:
всегда видно, куда ушли деньги, и почти всё можно поправить. Бэкенд уже умеет точный
откат визита (`revertVisit` в транзакции) и удаление абонементов/платежей с корректным
восстановлением (FK `SET NULL`) — задача оборачивает это в понятный журнал.

## 2. Скоуп (вариант C — деньги + ключевые действия с занятиями)

Логируются и обратимы:
- **Отметка ученика** (визит со списанием абонемента ИЛИ авто-платежом).
- **Оформление/продление абонемента** (создание `subscription`, опц. с платежом «оплачен сразу»).
- **Запись оплаты** (mark-paid: создание/привязка платежа к абонементу).
- **Создание занятия** — одиночное = одно событие; **серия** = по событию на каждое занятие,
  объединённые общим `batchId` (в ленте — группа с раскрытием, «Отменить всю серию»).

Логируется, но **view-only** (без отмены):
- **Удаление занятия** — честное восстановление (участники, визиты, деньги, событие
  Google-календаря) требует снимков и заметно усложняет фичу; решено НЕ делать обратимым.

Вне скоупа (не логируется): правки профиля ученика/группы, изменения локаций/тарифов,
смена настроек, переносы времени занятий (могут войти отдельным проектом «всё, что я делал»).

## 3. Поведение отмены

- **Гибрид (мгновенно + тост «Вернуть»).** Тап «Отменить» → событие сразу помечается
  отменённым + тост «Отменено · Вернуть» на несколько секунд (паттерн Gmail undo). «Вернуть»
  в тосте откатывает саму отмену (повторяет исходное действие). Модального подтверждения нет.
- **Без срока давности** — отменять можно когда угодно.
- **Идемпотентность / защита.** Повторная отмена уже отменённого = тихий no-op. Если объект
  события изменён/удалён вручную после события (например, абонемент уже удалён) — отмена
  делает то, что ещё возможно, и не падает; недостижимое — тихо пропускается.
- **Отменённое событие** остаётся в ленте зачёркнутым с меткой «Отменено» (видно, что
  поправил). Постоянной кнопки «повторить» в ленте нет — повтор = сделать действие заново
  обычным путём (или «Вернуть» в окне тоста).

## 4. Группировка массовых действий (вариант C — группа с раскрытием)

«Закрыть день» и быстрая отметка группы пишут события с общим `batchId`. В ленте такой
пакет — **сворачиваемая группа**:
- В свёрнутом виде: заголовок `Закрыт день · 8 отметок` (+ сводка денег) и действие
  **«Отменить всё»** (откатывает все дочерние события пакета).
- По тапу раскрывается в отдельные строки; **любую отметку можно отменить отдельно**.
- Одиночное действие (`batchId = null` или пакет из одного) показывается плоской строкой
  без группы.

«Отменить всё» = отмена каждого ещё не отменённого дочернего события пакета. Если часть
пакета уже отменена вручную — пропускаются.

## 5. Что показывает строка события

`<иконка> <главный текст> · <денежная деталь> — <время>` + действие «Отменить» (если обратимо):
- Отметка: `✅ Маша · «Старт» · списано занятие, осталось 3` либо `… · платёж 1 200 ₽`.
- Продление/оформление: `🔄 Вася · абонемент 8 занятий` (+ `· платёж 9 600 ₽`, если оплачен сразу).
- Запись оплаты: `💳 Петя · оплата 1 200 ₽`.
- Создание занятия: `➕ Создано занятие «Старт», 11:00`.
- Удаление занятия: `🗑 Удалено занятие «Старт», 11:00` (без «Отменить»).

Тексты — из денормализованного `summary` события (имена/суммы на момент действия), чтобы
лента не «ломалась», если объект позже удалят. UI-строки — через i18next.

## 6. Модель данных (бэкенд)

Новая таблица `activity_log` (миграция):

| Поле | Тип | Назначение |
|---|---|---|
| `id` | UUID PK | |
| `user_id` | UUID, индекс | владелец (тренер) |
| `type` | enum/string | `attendance_marked` · `subscription_created` · `payment_recorded` · `training_created` · `training_deleted` |
| `batch_id` | UUID null, индекс | объединяет события одного массового действия |
| `created_at` | datetime, индекс | порядок ленты (UTC, как везде в Sequelize) |
| `undone_at` | datetime null | признак и время отмены |
| `training_id` / `student_id` / `visit_id` / `subscription_id` / `payment_id` | UUID null | ссылки для отмены (без FK-каскадов — лог переживает удаление объектов) |
| `summary` | JSON | денормализованные поля для отображения (имена, суммы, кол-во занятий и т.п.) |

Снимков состояния НЕТ (единственное разрушительное действие — удаление — view-only).
Лента и выборка — по `(user_id, created_at)` индексу.

## 7. Бэкенд: запись и отмена

- **Запись** строки журнала — в той же транзакции, что и само действие, в точках:
  `markAttendance` (на каждого добавленного ученика), `removeAttendee` (как отмена — НЕ
  пишет новое событие, а ставит `undone_at` у исходного), создание абонемента
  (renew/issue), mark-paid, `createTraining`, `deleteTraining`. Запросы массовых действий
  принимают опциональный `batchId` из тела и прокидывают его в строки журнала.
- **Чтение:** `GET /activity-log?cursor=<created_at|id>&limit=N` — порциями, обратный
  хронопорядок, `{ data, nextCursor }`.
- **Отмена:** `POST /activity-log/:id/undo` — по `type` выполняет обратную операцию
  (переиспользует `revertVisit`, удаление абонемента/платежа/занятия), ставит `undone_at`,
  возвращает обновлённую строку. No-op при уже отменённом/недостижимом.
- **Отмена пакета:** `POST /activity-log/batch/:batchId/undo` — отменяет все ещё активные
  дочерние события (в транзакции), возвращает их обновлённые строки.
- Чистое ядро маппинга `type → обратная операция` выносится тестируемой функцией.

## 8. Фронтенд

- **Точка входа:** пункт **«Журнал»** в `app/layout/profile-menu.tsx` (рядом с «Настройки
  календаря»/«Тема») → роут `/journal` (ленивый, как остальные страницы в `routes.tsx`).
- **Экран** `pages/journal/`: лента, сгруппированная по дням («Сегодня», «Вчера», дата);
  внутри дня — события и сворачиваемые группы (`batchId`); infinite-scroll догрузка.
  Отменённые — зачёркнуты с меткой. Состояния loading/empty/error — существующие
  `list-skeleton`/`EmptyState`/`error-state`.
- **Модуль** `entities/activity-log/`: модель (типы события, маппинг тип→иконка/текст),
  API-репозиторий, хук `useActivityLog` (react-query `useInfiniteQuery`), мутации
  `undoEvent`/`undoBatch` с инвалидацией `activityLogKeys` + затронутых кэшей
  (`trainingKeys`/`studentKeys`/finance) — отмена меняет визиты/деньги.
- **Тост «Вернуть»** — существующий `useToast`. «Вернуть» = **повторно выполнить исходное
  действие** через единую мутацию `redo`, читающую `type` + `summary`-ссылки события
  (отметка → `markAttendance` заново; абонемент → создать заново; платёж → создать заново;
  занятие → создать заново). Повтор создаёт НОВУЮ запись (новый визит/платёж/абонемент) и
  пишет новое событие журнала; исходное остаётся помеченным «Отменено». Это однозначный,
  единообразный путь для всех обратимых типов.

## 9. Дефолты v1 (осознанно)

- **Без фильтров** (по ученику/типу) — только хронолента по дням. Фильтры — отдельный заход.
- **Хранить всё, без срока** — лог не чистится; объём контролируется пагинацией.
- **Только глобальный журнал** (история визитов в дровере ученика остаётся как есть, не
  дублируем undo туда в v1).

## 10. Тестирование (gate)

- **vitest (фронт, чистые функции):** группировка ленты по дням и по `batchId`; маппинг
  `type → иконка/текст`; формирование `summary`-строк.
- **jest (бэк):** откат каждого типа события; идемпотентность (двойная отмена = no-op);
  no-op при удалённом объекте; отмена пакета пропускает уже отменённые; запись `batchId`.
- **Сборки:** `npm run build -w apps/frontend` и `-w apps/backend` — 0 ошибок.
- **eslint:** 0/0.
- **Ручная проверка (демо):** отметить ученика → событие в журнале → «Отменить» →
  списание/платёж откатились (видно в Финансах); «Закрыть день» → группа с раскрытием →
  «Отменить всё» и отмена одной строки; продление с оплатой → отмена удаляет абонемент и
  платёж; удаление занятия → строка без «Отменить»; пункт «Журнал» в меню профиля
  (десктоп+мобайл).

## 11. Ловушки

- **~6 точек записи** — пропустить хоть одну = дыра в журнале. Точки перечислены в §7,
  в плане каждая — отдельная задача с тестом.
- **`training.groupId` на фронте = ИМЯ группы** (не UUID) — для `summary` брать имя через
  существующий маппинг.
- **UTC** — `created_at` пишется в UTC (Sequelize); группировка по дням на фронте — по
  локальной дате пользователя.
- **Инвалидация кэшей** после отмены — отмена меняет визиты/абонементы/платежи; не
  инвалидировать = устаревший UI на других экранах (как ловушка из «Мой день»).
- **`batchId` генерит фронт** один на пользовательское массовое действие («Закрыть день»/
  быстрая отметка группы) и шлёт в каждый вызов мутации пакета.
